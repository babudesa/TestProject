Adding a High-Risk Indicator
You can create a new high-risk indicator. The general steps are that you must create the icon, subtype ClaimIndicator, and add the implementation code.
WARNING Guidewire strongly recommends limiting the number of indicators used for any one line of business. Overuse of indicators lessens the overall impact to end users. Additionally, Guidewire designed the Claim Summary screen with the expectation that few, if any, claims will have more than four or five indicators. If the number of indicators per claim exceeds this expectation, then Guidewire recommends that you revisit the Claim Summary screen design and determine if it needs to be modified. Otherwise, important claim information can appear further down the screen necessitating additional scrolling.
Note: Each time you create a new subtype, you must modify the .pcf files so that the new indicator can display in the info bar and on the Claim Status screen. While it also appears on the Claim Summary screen, that screen relies on the generic indicator interface providing your indicator implements the standard On property and has an icon. If it does, then it shows in the screen automatically, whenever it is on.
To create the high-risk icon
1.
Create the image using a third-party graphics program.
2.
In Guidewire Studio, navigate to the Web Resources → resources → Ocean → images. Right-click and select New → Other file. Ensure the file name is the same as the name of the file created in the third-party graphics program.
3.
Move or copy the third-party graphics image file into the install directory. The path is module → configuration → webresources → Ocean → images so that it replaces the file created by Studio.
4.
If the server is running, stop it.
5.
Execute gwcc dev-deploy in the Command window.
6.
Restart the server.
To add a new subtype in Studio
Perform the following steps in Guidewire Studio.
1.
Your selection depends on the type of indicator that you want. Your new subtype can look like the following example: 
<?xml version="1.0"?>
<subtype xmlns="http://guidewire.com/datamodel" desc="Litigation" 
   entity="LitigationClaimIndicator" final="false" priority="1" 
   supertype="ClaimIndicator">
  <implementsInterface iface="gw.api.claim.indicator.ClaimIndicatorMethods"
  impl="gw.claim.indicator.LitigationClaimIndicatorMethodsImpl"/>
</subtype>
The implementsInterface element must be present as seen in this example except that the impl class is your own implementation class. The iface attribute defines the interface that the subtype implements.
Use the package gw.claim.indicator and your new name<subtype>MethodsImpl class.
2.
Adding the new subtype automatically adds a new typecode to the list and you are configuring the secondary attributes (name, description, priority) of this automatically created typecode.
You must extend the ClaimCenter data model by adding a new subtype of the entity ClaimIndicator. In Studio, navigate to configuration → Typelists → ClaimIndicator to see the following subclasses that are listed under the Code column:
•
ClaimIndicator
•
CoverageInQuestionClaimIndicator
•
FatalityClaimIndicator
•
FlagClaimIndicator
•
LargeLossClaimIndicator
•
LitigationClaimIndicator
•
SIUClaimIndicator
3.
Edit the typecode as seen in the following example:
<typecode code="ExampleClaimIndicator" desc="Example Claim Indicator" name="Example" priority="5"/>
The name used in this typecode element is the name that appears in the user interface for this type of indicator. This step allows you to set up a (possibly localized) name and priority for the indicator. The priority is used when displaying the indicators to decide the order in which they appear.
4.
Implement your new <subtype>MethodsImpl class. You could make your class implement ClaimIndicatorMethods directly but subclassing the ClaimIndicatorMethodsImpl class provides a couple of simple conveniences:
•
It handles the icon. You only need to specify the string name of your indicator icon when you call the constructor of ClaimIndicatorMethodsImpl.
•
It provides a method called setOn(newValue : boolean), which can be used to set the IsOn flag and WhenOn field of the indicator.
The following is an example implementation:
class LitigationClaimIndicatorMethodsImpl extends ClaimIndicatorMethodsImpl {
 
  /**
   * Constructor, called when an indicator is created or read from the database
   */
  construct(inIndicator : LitigationClaimIndicator) {
    super(inIndicator, "indicator_icon_litigation.gif") // Passes in name of indicator icon
  }
 
  /**
   * Update, sets the indicator on if the the claim litigation status is "litigated"
   * or "complete"
   */
  override function updateMetricValue() {
    var status = Indicator.Claim.LitigationStatus
    setOn(status == "litigated" or status == "complete") 
   // Calls setOn so that both IsOn and WhenOn fields will be set
  }
 
  /**
   * Text label, returns the description of the current claim litigation status
   */  
  override property get Text() : String {
    return Indicator.Claim.LitigationStatus.Description
  }
 
  /**
   * Hover text, returns the names of any open matters or a special label if there
   * are none
   */
  override property get HoverText() : String {
    var openMatters = Indicator.Claim.Matters.where(\ m -> not m.Closed).orderBy(\ m -> m.CreateTime)
    return openMatters.Count > 0
      ? openMatters.map(\ m ->
      m.Name).join(displaykey.Web.LitigationClaimIndicator.MatterNameSeparator)
      : Indicator.Claim.LitigationStatus.DisplayName
  }
  
}
5.
Add a new info bar element to the ClaimInfoBar.pcf file. This has to be done explicitly even though the required element is standard. Assuming your new indicator type is called ExampleIndicator, the new element appears as the following:
 
<InfoBarElement
      icon="Claim.ExampleIndicator.Icon"
      id="ExampleIndicator"
      tooltip="Claim.ExampleIndicator.HoverText"
      visible="Claim.ExampleIndicator.IsOn"/>
6.
Add a new modal input set called ClaimIndicatorInputSet.ExampleIndicator.pcf. This is used by the Claim Status screen to display the details of your indicator. The Claim Status screen iterates through all indicators on the claim and displays an input set for each one. It handles the indicator as a require argument to the input set. You can put whatever you want in the input set. It is helpful to follow a similar style to the input sets for the existing indicators which tend to be fairly small. If you make your input set too large it can drastically change the layout of the Claim Status screen.
Note: ClaimCenter displays the indicators by subtype order. So an indicator whose subtype has priority 1 appears before an indicator whose subtype has priority 3.
