ClaimCenter Configuration Guide : Data Model Configuration : The Domain Graph : Working with Changes to the Data Model
Working with Changes to the Data Model
As the Guidewire ClaimCenter server starts, it automatically runs the database upgrade utility and makes any changes to the ClaimCenter database mandated by the upgrade utility. Upon successful completion of the database upgrade, ClaimCenter checks to see if the database schema matches the existing data model.
•
If the two match, server start up continues without any issues.
•
If the two do not match, then the server fails with an error.
However, many changes that you make to the data model that reference tables used in the archiving process can involve data that you need to move manually between tables. You must correct these issues before you can start the ClaimCenter server. This is true only if the change causes a database table to move from one graph to another (from the domain graph to the admin graph, for example). Guidewire refers to this table movement as a transition.
Sequence of Events
The following sequence of events takes place if you modify the data model and the modification involves any tables referenced by the archiving process:
1.
You modify the data model and start the application server.
2.
The server performs a database upgrade and verifies the existing data model against the ClaimCenter database:
•
If the database schema matches the data model, the server completes its start up without error.
•
If configuration parameter ArchiveEnabled is set to true and the server detects an issue with table data that requires human intervention, it fails with an error. In this case, you must continue and complete the following steps.
3.
The server populates the ArchiveTransitionRecord table with one row for each transition that it detects. As it creates each row, the server sets the Completed and Accepted columns on that row to false.
4.
You fix all of the listed issues and set the Completed column on table ArchiveTransitionRecord to true. The server will continue to fail start up with the same error message until you set value of all the Completed columns in the ArchiveTransitionRecord table to true.
5.
You restart the server. It now determines that the database schema and the data model match and it completes the start up process.
6.
The server sets the value of the Accepted column on table ArchiveTransitionRecord to true.
Correcting Error Conditions
You can divide all database tables into one of the following groups:
•
The tables in the Domain (D) graph
•
The tables in the Admin (A) graph
•
The Overlap (O) tables
•
The rest (R) of the tables in the database
The following sections describe what happens if you modify the data model in such a way as to cause a table to move (transition) from one group to another. The information makes the following assumptions:
•
You have already archived multiple claims across multiple archive databases.
•
You have fully synchronized the admin data before starting the transition move.
Some of the following transitions require human intervention. Other transitions, ClaimCenter can handle automatically. ClaimCenter does not generate an error for transitions that it can handle automatically. If you receive a transition error on server start up, you must correct the data issue before you can continue and start the application.
From Domain (D) → To...
 
Admin (A)
Human intervention required
This transition can occur, for example, if you decide that a table actually contains shared data and decide to move the table to the admin graph.
ClaimCenter does not archive the admin graph. Instead, ClaimCenter synchronizes the admin graph with the main application database. In other words, ClaimCenter mirrors the admin graph from the main database into one or more archive databases. Thus, the table undergoing transition can be spread out across multiple databases.
To correct a transition error of this kind, you must recombine the data by copying the contents of the table in each archive database into the main database. In this type of operation, there is a possibility of duplicate rows as you recombine the data, which can violate unique index constraints.
Overlap (O)
No action required
This transition can occur, for example, if you attempt to correct a previous mistake.
If you are moving a table from the domain group to the overlap group, there is really nothing that you need do. In an overlap table, a row can belong to either the domain graph or to the admin graph, but not to both. Thus, if there is a row in the domain table that belongs in the admin table, the next admin table synchronization moves it to the correct location.
Rest (R)
Human intervention required
This transition can occur, for example, if you realize that a table is shared and decided to link to it off ClaimInfo.
As in moving a table from the domain group to the admin group, the contents of the table that you want to move are spread out across multiple databases. This requires that you recombine data stored in multiple archive tables into a single table in the primary database. You then need to delete the data rows from the archive database or databases in which it previously existed.
You must also ensure that all affected rows are referenced from the appropriate ClaimInfo row. Again, there is a possibility of duplicate rows as you recombine the data, which can violate unique index constraints.
From Admin (A) → To...
 
Domain (D)
Human intervention required
This transition can occur, for example, if you discover a mistake and decide that the relationship between a claim and this entity is actually that of one-to-one.
In the simplest case, you need merely delete any non-referenced rows from that table in each database. (A non-referenced row is one to which no foreign key points and one that does not have a foreign key in any row.)
However, if there are rows that are shared, then a subsequent archive or restore could fail:
•
A subsequent archive operation could possibly violate the unique index constraint.
•
A restore operation could possibly violate a foreign key constraint.
It is up to you to be aware of these possibilities and to plan your data model change accordingly.
Overlap (O)
Human intervention required
This transition can occur, for example, if you made an initial mistake of the table placement into the admin group.
While all the data for that table exists in each database, you must also do the following:
•
You must determine to which graph a row belongs.
•
You must delete the rows that do not belong to the admin graph and do not belong to any claim presently in that archive database.
In most cases, it is sufficient to delete all non-referenced rows.
Again, there is also the possibility a subsequent archive or restore could fail if there is shared data.
Rest (R)
Human intervention required
This transition can occur, for example, if you decides that a table changes too frequently. Thus, it is not necessary to mirror it in the archive databases (to the user statistics table).
In this case, manually truncate the table in each archive database as there could be a foreign key to that table. ClaimCenter automatically deletes the related rows in the ArchiveAdminKey table in the main database (which contains the key mapping).
From Overlap (O) → To...
 
Domain (D)
No action required
This transition can occur, for example, in the course of correcting an initial mistake. If moving an overlap table into the domain graph, all rows in the table are most likely already part of the domain graph. In this case, all rows in the table are already part of some domain graph instance and therefore were moved during an archive of the domain graph. This means that there are no non-owned rows in any database. If this is the case, you need to do nothing further.
Admin (A)
No action required
This transition occur, for example, in the course of correcting an initial mistake. All rows in the overlap table most likely already exist as part the of admin graph. If this is the case, you need to do nothing further.
Rest (R)
Human intervention required
This transition can occur, for example, in the course of correcting an initial mistake. Perhaps, in a similar manner to a Domain (D) → Rest (R) transition, the table is now linked off of ClaimInfo.
As with the Domain (D) → Admin (A) transition and Domain (D) → Rest (R) transition, the contents of the table are spread out across multiple databases. You need to recombine the data.
•
First, combine the data into a table in the primary database, and, if needed, reference the rows from the appropriate ClaimInfo row.
•
Secondly, delete the data from the archive database.
Again, there is a possibility of duplicate rows as you recombine the data, which can violate unique index constraints.
From Rest (R) → To...
 
Domain (D)
No action required
This transition can occur, for example, if you discover that a table really belongs in the domain graph. If this is the case, there are no archived claims that are dependant on the data in this table. As such, there is nothing further that you need do.
Admin (A)
No action required
This transition can occur, for example, if you discover that an entity did not properly implement the AdminTable delegate. (For example, it was inadvertently left off the entity definition, somehow.) If this is the case, there are no archived claims that are dependant on the data in this table. As such, there is nothing further that you need do. The next synchronization of the admin table will move the data to the archive database.
Overlap (O)
No action required
This transition can occur, for example, for the same reasons as the Rest (R) → Domain (D) transition and the Rest (R) → Admin (A) transition. Again, there is nothing further that you need do.
