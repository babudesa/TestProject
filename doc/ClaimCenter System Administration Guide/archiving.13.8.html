Generating Database Statistics on Archive Databases
Keep database statistics current on the primary archive database. Especially be sure to generate statistics after first populating the archive or archiving a large number of records. Failure to generate statistics on the archive database can lead to slow performance.
IMPORTANT If you run database consistency checks against the archive database, you might see warnings that a policy must have a PeriodPolicy if all its period fields are not null. These warnings are expected because aggregate limit-related data are not available in the archive database.
SQL Server keeps statistics current automatically if Auto Create Statistics is set to true, the default. Oracle database statistics must be updated through database commands.
The archive work queue does not explicitly update database statistics.
There are two methods that can be used to update the database statistics on the archive database:
•
Start an archive server on the target archive database. See "Using an Archive Database as the Main Database". Then use the maintenance tools to either update the statistics directly (-startprocess dbstats), or get the SQL statements to update statistics and run them manually (-getdbstatisticsstatements).
•
Running in normal, non-archive server mode, use the maintenance tools to get SQL statements to update statistics on the main database (-getdbstatisticsstatements). Then, edit the SQL statements to point to the archive database, and run them manually.
