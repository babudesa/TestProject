ClaimCenter System Administration Guide : Archiving Claims : Understanding Claim Archiving
Understanding Claim Archiving
By default, ClaimCenter uses just one condition to determine whether to archive a claim: how long the claim has been closed. The config.xml parameter DaysClosedBeforeArchive defines how long after a claim is closed that ClaimCenter archives the claim.
You can write archiving rules that reflect your unique business conditions. For example:
•
Do not archive a sensitive claim, or one with restricted access control.
•
Do not archive a claim that meets a certain condition, such as having medical payments over some amount.
•
Do not archive a claim whose claimant has other open claims pending.
Several methods, such as hasReportedArchiveProblem() and claim.reportArchiveProblem() are useful in rule writing. See the sample rules included in the archive rule set for usage details.
For more information on archiving rules, see “Archive” in the Rules Guide.

The Archive Work Queue
The archive work queue selects claims for archiving and moves eligible claims to the archive database.
The work queue performs a database query to identify potential claims in the active database for archiving. A potential claim must have been closed for at least DaysClosedBeforeArchive days, have no active messages, and not be part of an aggregate limit. Potential claims also must not have been previously excluded by a worker.
The work queue then applies the archiving rules to each potentially eligible claim and either archives the claim or excludes the claim from archiving.
To archive a claim, the archive work queue:
•
synchronizes administrative data and typekeys between the main repository and the archive.
•
copies all rows to the archive database, table by table.
•
relinks all the foreign keys of the claim.
•
deletes the copied row from the main database.
•
updates a ClaimInfo object for the claim, adding all needed references to cross-claim objects, like associations and bulk invoices.
•
creates a ClaimChanged history event.
WARNING Do not change the name of the archive after you have run the archive work queue. The archive name must be the same to restore archived claims to the main repository.
After running the archive work queue, Guidewire recommends that you update database statistics on both the active and primary archive databases. The archive process makes large changes to the tables. Updating database statistics enables the optimizer to pick better queries based on more current data. See "Generating Database Statistics on Archive Databases" and "Configuring Database Statistic Generation".
Archive Events and Histories
Every archived claim generates a ClaimInfo event with an ArchiveState of archived. Restoring a claim generates a similar ClaimInfo event with an ArchiveState of restored. In default mode, ClaimCenter does not use or refer to these events. In addition, a claim History receives a new entry each time ClaimCenter archives or restores the claim. Archive rules that reference these history entries or a ClaimInfo event can distinguish archived and restored claims from other claims.
The ClaimInfo Entity
As ClaimCenter creates a claim, it also creates a ClaimInfo entity. The ClaimInfo is a stub that remains in the active database after ClaimCenter archives the claim. Simple searches occur only in the active database, using claim data to find active claims, and comparable ClaimInfo data to find archived claims. ClaimCenter uses the ClaimInfo entity to locate claims efficiently by avoiding a search of both databases.
A ClaimInfo entity also contains links to bulk invoices and claim associations, so that a restored claim remains connected to these multi-claim entities, which are always in the main database.
The stub also contains the ArchiveState of the claim. The ArchiveState, from the ArchiveState typelist, can be archived or retrieving. A null ArchiveState indicates that the claim is in the active database, either because it has never been archived or has been successfully restored from the archive.
